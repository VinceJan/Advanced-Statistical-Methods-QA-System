# 部署方案

本项目已完成本地可运行版本，并已部署到 VPS。本文档同时记录当前线上部署方式，以及以后迁移、回滚和运维时应遵守的边界。

日常维护、日志查看、备份、发布、回滚和故障排查请优先阅读 `docs/运维手册.md`；本文更偏部署方案选择和线上状态说明。

## 推荐结论

正式部署建议使用 Docker Compose。

原因：

- 后端依赖 Python、前端依赖 Node，直接裸机部署容易受服务器环境影响。
- 项目依赖教材 PDF、SQLite 数据库和 `.env`，容器化更容易明确挂载和备份边界。
- 以后如果从 SQLite 迁移到 PostgreSQL、从本地检索迁移到 Chroma/FAISS，也更容易扩展。

如果只是临时在个人服务器上演示，也可以先用“裸机部署 + Nginx/Caddy 反代”。但长期交付、答辩展示或多人访问，推荐 Docker Compose。

1. 已轮换或确认 MiniMax API Key 安全。
2. 已修改默认管理员密码。

## 方案 A：裸机部署

适合：快速演示、个人服务器、短期运行。

### 服务器依赖

- Python 3.12 或兼容版本。
- Node.js 20+ 和 npm。
- Nginx。
- PowerShell 脚本需要改写为 Linux shell，或在 Windows Server 上直接使用现有脚本。

### 基本流程

1. 上传项目目录到服务器，例如 `/opt/stat-learning-qa`。
2. 放置教材 PDF：`/opt/stat-learning-qa/ISLRv2_corrected_June_2023.pdf`。
3. 创建 `.env`：

```env
MINIMAX_BASE_URL=https://api.minimaxi.com/v1
MINIMAX_MODEL=MiniMax-M3
MINIMAX_API_KEY=真实 Key
APP_BOOTSTRAP_INDEX=true
APP_DATABASE_URL=sqlite:///data/app.db
APP_PDF_PATH=ISLRv2_corrected_June_2023.pdf
ADMIN_USERNAME=admin
ADMIN_PASSWORD=改成强密码
```

4. 安装后端依赖：

```bash
python -m venv .venv
./.venv/bin/python -m pip install --upgrade pip
./.venv/bin/python -m pip install -r backend/requirements.txt
```

5. 安装并构建前端：

```bash
cd frontend
npm ci
npm run build
cd ..
```

6. 启动后端：

```bash
PYTHONPATH=. ./.venv/bin/python -m uvicorn backend.app.main:app --host 127.0.0.1 --port 8000
```

7. 用 Nginx 托管 `frontend/dist`，并把 `/api/` 反代到 `127.0.0.1:8000`。

### Nginx 示例

```nginx
server {
    listen 80;
    server_name your-domain.example;

    root /opt/stat-learning-qa/frontend/dist;
    index index.html;

    location / {
        try_files $uri /index.html;
    }

    location /api/ {
        proxy_pass http://127.0.0.1:8000/api/;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}
```

生产环境应再接入 Certbot 或其他证书方案启用 HTTPS。

## 方案 B：Docker Compose

适合：正式部署、可迁移演示、服务器环境不想手工污染。

### 建议服务划分

- `backend`：FastAPI + SQLite + PDF 索引。
- `frontend`：Nginx 托管静态前端。
- `caddy`：仅在使用 `docker-compose.yml` 时启用，负责 HTTPS 和反代。
- 卷挂载：
  - `./data:/app/data`
  - `./ISLRv2_corrected_June_2023.pdf:/app/ISLRv2_corrected_June_2023.pdf:ro`
  - `.env` 通过 `env_file` 注入后端。

### 镜像规划

后端镜像：

- 基础镜像：`python:3.12-slim`
- 工作目录：`/app`
- 安装：`backend/requirements.txt`
- 启动：`uvicorn backend.app.main:app --host 0.0.0.0 --port 8000`

前端镜像：

- 构建阶段：`node:20-alpine` 执行 `npm ci && npm run build`
- 运行阶段：`nginx:alpine`
- 静态文件：复制 `frontend/dist` 到 Nginx html 目录
- Nginx：把 `/api/` 反代到 `backend:8000`

### Compose 文件选择

`docker-compose.yml` 适合服务器没有现成 Web 入口时使用：

```bash
docker compose -p advanced-stat-qa -f docker-compose.yml up -d --build
```

`docker-compose.server.yml` 适合服务器已有 Caddy/Nginx 时使用，也是当前线上采用的方式：

```bash
docker compose -p advanced-stat-qa -f docker-compose.server.yml up -d --build
```

当前线上反代由宿主机 Caddy 负责，不启用 Compose 内置 Caddy 服务，避免和已有 80/443 端口冲突。

### 线上发布流程

#### 自动部署（推荐）

本项目已配置 GitHub Actions 自动部署。工作流定义在 `.github/workflows/deploy.yml`：

- **触发条件**：`push` 到 `main` 分支
- **执行步骤**：
  1. 检出代码
  2. 设置 SSH 密钥（从 GitHub Secrets 读取）
  3. 打包项目（排除 `.env`、`.git`、`node_modules` 等）
  4. 通过 SSH 上传到 VPS
  5. 在 VPS 上停止旧容器、备份旧版本、解压新版本
  6. 恢复 `.env` 环境变量
  7. 构建并启动新容器
  8. 执行健康检查

**本地只需执行**：

```powershell
.\scripts\verify.ps1
git add .
git commit -m "说明本次修改"
git push
```

GitHub Actions 会自动完成后续部署。

**配置要求**：
- GitHub 仓库 Settings → Secrets → 添加 `VPS_SSH_KEY`（SSH 私钥）
- GitHub 仓库 Settings → Secrets → 添加 `VPS_HOST`（VPS 域名或 IP）
- VPS 上已配置对应公钥到 `~/.ssh/authorized_keys`

#### 手动部署（备用）

如果自动部署不可用，按以下步骤手动发布：

1. 在本地确认 `.\scripts\verify.ps1` 通过。
2. 打包源码，不包含 `.env`、`.venv/`、`data/`、`frontend/node_modules/`、`frontend/dist/`。
3. 上传源码包到服务器 `/tmp/advanced-stat-qa.tar.gz`。
4. 单独上传服务器 `.env`，权限设为 `600`。
5. 解压到 `/opt/advanced-stat-qa/app`。
6. 执行：

```bash
cd /opt/advanced-stat-qa/app
docker compose -p advanced-stat-qa -f docker-compose.server.yml up -d --build
```

7. 验证：

```bash
curl -fsS http://127.0.0.1:18000/api/health
curl -fsS https://aistudyassistant.bluesclawd.dev/api/health
```

8. 用浏览器登录公网地址，检查知识图谱、问答、历史和管理后台。

## 数据持久化与备份

SQLite 数据库默认在 `data/app.db`，包含：

- 用户和登录 token。
- 会话与消息。
- 学习历史。
- 问答对。
- 知识点和图谱边。
- 教材文本块。
- 参考书元数据、上传 PDF 和本地向量索引。

部署时必须持久化 `data/`。备份最简单方式是停服务后复制：

```bash
cp -r data data-backup-$(date +%F)
```

如果未来多人长期使用，建议迁移到 PostgreSQL，并把文本索引重建改为后台任务。

## 性能与扩展

当前性能特征：

- 热缓存后 RAG 检索通常很快。
- `APP_RETRIEVAL_MODE=auto` 时，向量索引可用则走本地向量检索；索引缺失时自动降级到 TF-IDF。
- 主要耗时通常来自 MiniMax LLM 生成。
- PDF 索引构建发生在首次启动或重建索引时。

可选优化：

- 引入 Chroma/FAISS，替换或增强当前本地混合检索。
- 将当前轻量哈希向量索引升级为 FAISS/Chroma 时，仍需保留 `tfidf` fallback 和 `auto` 降级模式。
- 将索引重建放入后台任务队列。
- 为 MiniMax 调用增加 token、耗时、错误率统计。
- 若多人并发，优先迁移 PostgreSQL，再考虑缓存服务。

## 安全清单

- 不要把 `.env` 放进镜像层或提交记录。
- 公网部署前修改 `ADMIN_PASSWORD`。
- 公网部署必须 HTTPS。
- 限制服务器防火墙，只暴露 80/443。
- 定期备份 `data/`。
- `data/reference_books/` 和 `data/vector_index/` 属于运行数据，和 SQLite 数据库一起备份。
- 若 Key 曾经出现在聊天、截图或仓库历史中，应在 MiniMax 控制台轮换。
