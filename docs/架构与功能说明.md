# 架构与功能说明

本文面向第一次接手项目的开发者或 AI Agent，说明系统由哪些模块组成、核心功能如何工作、数据如何流转，以及修改时最容易破坏的边界。

## 一句话定位

这是一个面向“高级统计方法”课程的中文智能问答系统。系统以英文 `ISLRv2_corrected_June_2023.pdf` 为 RAG 主语料，结合课程知识图谱和 MiniMax LLM，为用户提供有来源、有边界、可审计的课程问答。

## 技术栈

| 层 | 技术 |
| --- | --- |
| 前端 | React、TypeScript、Vite、Cytoscape.js、React Markdown、KaTeX、lucide-react |
| 后端 | FastAPI、SQLAlchemy、SQLite、pypdf、httpx |
| LLM | MiniMax OpenAI-compatible Chat Completions |
| 部署 | Docker Compose、Nginx 前端容器、FastAPI 后端容器、宿主机 Caddy |
| 验证 | pytest、Vite build、npm audit、Playwright |

## 问答流程

一次正常问答大致如下：

```text
前端提交问题
  -> POST /api/chat/ask
  -> RagService.get_or_create_conversation
  -> recent_messages 取最近上下文
  -> contextualize_question 补全追问
  -> analyze_question
      -> find_concepts 匹配知识点
      -> retrieve 检索教材文本块
      -> 判断 out_of_scope / insufficient_evidence / answered
  -> get_subgraph 获取相关知识图谱
  -> recommended_questions 生成推荐问题
  -> LLMClient.answer 调用 MiniMax 或本地降级
  -> 保存 ChatMessage 和 QuestionHistory
  -> 前端展示回答、来源、性能、推荐问题和图谱
```

## 可信度边界

系统不是“什么都答”的聊天机器人。以下边界必须保留：

- 无关问题返回 `out_of_scope`。
- 无关问题必须满足：
  - `sources=[]`
  - `related_questions=[]`
  - `graph.nodes=[]`
  - `graph.edges=[]`
- 统计相关但证据不足的问题返回 `insufficient_evidence`。
- 证据不足时不能编造教材页码和来源。
- 清晰无关问题不能被多轮上下文污染，例如连续追问后再问“我是谁”仍应拒答。
- 每一轮追问都重新计算来源、推荐问题和图谱，不能沿用上一轮证据。

这些逻辑主要在 `backend/app/rag.py`，测试在 `backend/tests/test_app.py`。

## RAG 检索

当前检索不是外部向量数据库，而是本地混合检索：

- PDF 抽取正文进入 `text_chunks`。
- 中文问题通过 `TERM_EXPANSIONS` 扩展英文关键词。
- 文本块分词、文档频率和词频放入进程内缓存。
- 检索返回 `SourceOut`，含：
  - `chunk_id`
  - `chapter`
  - `section`
  - `pdf_page`
  - `source_file`
  - `snippet`
  - `summary`
  - `score`

清洗策略包括：

- 过滤目录/前言噪声。
- 过滤图注、坐标轴、过密数字和公式碎片。
- 来源侧展示可读摘要，不直接展示大段 PDF 抽取原文。

未来如迁移 Chroma/FAISS，必须保留现有 `out_of_scope` 和 `insufficient_evidence` 守门逻辑。

## 知识图谱

知识图谱包含：

- 概念节点：`Concept`
- 关系边：`GraphEdge`
- 子图查询：`get_subgraph`
- 推荐问题：`recommended_questions`

前端使用 Cytoscape.js 展示图谱。不要恢复手写 SVG 图谱布局，因为之前出现过节点文字重叠、英文截断和排布不稳定问题。

## LLM 配置

配置来源分两层：

1. `.env`：启动默认值。
2. `system_config` 表：管理员后台保存的运行时配置。

运行时读取逻辑在 `backend/app/app_config.py` 和 `backend/app/llm.py`。

管理员接口不会返回完整 API Key，只返回：

- `has_api_key`
- `api_key_preview`
- `base_url`
- `model`
- `disabled`

## 修改时的高风险点

### 1. RAG 范围判断

修改 `backend/app/rag.py` 时，必须测试：

- `我是谁`
- `今天天气怎么样`
- `岭回归和 Lasso 有什么区别？`
- `那它为什么能做变量选择？`

### 2. 管理员权限

修改 `backend/app/routers/admin.py` 或前端后台时，必须测试：

- 普通学生看不到后台入口。
- 普通学生请求 `/api/admin/users` 返回 403。
- 管理员可正常打开后台。

### 3. 图谱显示

修改图谱时，必须测试：

- 学生端打开知识图谱页。
- 搜索“岭回归”。
- 概念列表英文完整显示。
- 图谱不空白、不重叠。

### 4. 部署配置

修改 Docker 或端口时，必须同步：

- `docker-compose.yml`
- `docker-compose.server.yml`
- `docs/运维手册.md`
- `docs/部署方案.md`

并确认 Caddy 仍反代到正确端口。

## 性能与并发设计

系统部署在弱配置 VPS 上，已做以下优化以承载多用户并发：

- **SQLite WAL 模式**：`backend/app/database.py` 在每个连接上设置 `PRAGMA journal_mode=WAL`、`busy_timeout=5000`、`synchronous=NORMAL`，允许读写并发，避免写锁冲突立即报错。
- **LLM 调用并发控制**：`backend/app/llm.py` 使用 `asyncio.Semaphore(5)` 限制同时进行的 LLM 调用为 5 个，防止并发请求堆积导致内存暴涨。
- **Uvicorn 请求限制**：`Dockerfile.backend` 启动参数包含 `--limit-concurrency 30` 和 `--timeout-keep-alive 30`，超过 30 个并发请求时返回 503，防止 OOM。
- **Docker 内存限制**：`docker-compose.server.yml` 限制 backend 容器 512M、frontend 容器 64M，防止单个容器吃光 VPS 内存。
- **Nginx 静态资源优化**：`frontend/nginx.conf` 启用 gzip 压缩，带 hash 的 Vite 产物长期缓存（1 年），减少带宽和重复请求。
- **Caddy 反代超时**：`deploy/Caddyfile` 设置 60s 读写超时，适配 LLM 长耗时调用。

这些优化使系统在 1-2 核 / 1-2G 内存的 VPS 上可承载约 15-25 人同时提问。前端为纯静态页面，浏览器保持打开不消耗服务器资源，只有主动操作才产生请求。
