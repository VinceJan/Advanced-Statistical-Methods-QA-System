# 题目二项目目标约束

本文用于启动后续目标模式：先锁定项目范围、关键技术决策、验收标准和待确认事项。若本文与 `题目2.md` 的硬性要求冲突，以 `题目2.md` 为最低交付底线；本文用于把成果做得更完整、更稳、更像可正式演示上线的系统。

## 已读输入

- `题目2.md`：要求实现基于高级统计方法课程知识图谱的智能问答系统，包含问答对管理、RAG 问答、知识图谱辅助问答、用户管理、源码和实验报告。
- `ISLRv2_corrected_June_2023.pdf`：英文教材，615 页；抽样和全量文本检查显示 611 页可直接抽取正文，关键词如 cross-validation、ridge regression、logistic regression、Lasso 等可稳定命中。
- `[BOOK]统计学习导论 基于R应用-高清中文.pdf`：中文教材，323 页；抽样页均为图片型页面，`text_len=0`、`fonts=0`、`xobjects=1 /Image`，当前无法直接抽取正文。

## 总体目标

交付一个中文使用体验的课程级完整 Web 系统：学生用中文提问，系统结合教材文本、知识图谱和已整理问答对，给出中文回答、来源引用、相关概念推荐、图谱子图和个人历史记录。

目标不是只凑够题目功能，而是按评分项倒推，做成可演示、可复现、可维护、可写进实验报告的完整成果。

## 教材选择约束

推荐默认使用 `ISLRv2_corrected_June_2023.pdf` 作为 RAG 主语料，系统界面、问答对、回答内容和报告均使用中文。

理由：

- 英文 PDF 是文本型 PDF，可直接抽取、分块、建立页码引用；中文 PDF 是扫描/图片型 PDF，若直接用于 RAG，必须先 OCR。
- OCR 会额外引入识别错误、术语断裂、公式污染和页码对齐问题；这些问题会直接影响 RAG 检索效果和来源引用可信度。
- 英文教材的统计学习术语更标准，适合作为事实源；中文体验可通过中文问题理解、双语术语表、跨语言向量模型和中文生成回答解决。
- 在中文语境下，真正重要的是“用户输入与输出中文”，不一定要求底层语料本身中文。只要检索层支持中英术语对齐，英文主语料反而更稳。

如果后续坚持使用中文教材，则必须新增 OCR 阶段，并把 OCR 后的文本作为新的可验证语料；这会显著增加工期和质量风险。除非课程老师明确要求中文教材，否则不建议走 OCR 路线。

## 必做功能边界

系统至少覆盖以下功能：

- 问答对数据管理：不少于 50 个高质量问答对，覆盖概念解释类、关系查询类、应用场景类，并支持增删改查、搜索、类型筛选、知识点筛选。
- RAG 问答：从教材 PDF 抽取文本，按章节和页码分块，建立至少 50 个文本块的向量库；每次问题检索 3-5 个相关块；回答末尾展示教材章节、页码等来源。
- 知识图谱辅助问答：支持关系查询、图谱路径或邻域查询、点击概念节点后推荐相关问题、回答页展示核心概念子图。
- 用户管理：注册、登录、登录状态、每个用户的提问历史、历史查看和清空。
- 实验报告：说明需求、架构、数据构建、RAG 流程、知识图谱设计、测试样例、运行方式和效果分析。

为提高质量，目标实现应高于最低线：

- 问答对目标数量：60-80 个，而不是刚好 50 个。
- 知识点目标数量：60 个以上核心概念。
- 图谱边目标数量：100 条以上，边类型包含属于章节、相关、区别、用于、前置、评估方法、正则化方法等。
- 文本块目标数量：不少于 200 个，并保留 chapter、section、pdf_page、source_file 等元数据。

## 推荐工程方案

默认采用“FastAPI 后端 + React 前端 + SQLite 持久化 + Chroma 向量库 + NetworkX 图查询”的方案。

后端：

- Python FastAPI 提供 API。
- SQLite + SQLAlchemy 存储用户、问答对、知识点、图谱边、提问历史、文本块元数据。
- Chroma 作为本地持久化向量库；若依赖冲突，再降级为 FAISS + SQLite 元数据表。
- NetworkX 用于图谱邻域、路径和关系查询；不默认使用 Neo4j。

前端：

- React + TypeScript + Vite。
- 图谱可视化使用 Cytoscape.js 或 React Flow，优先选择适合节点/边关系展示的方案。
- 首屏直接是可用的问答工作台，不做营销式 landing page。

模型与检索：

- 首选多语言向量模型 `BAAI/bge-m3`，用于中文问题和英文教材之间的跨语言检索。
- 若本机下载或性能不稳定，可降级为 `intfloat/multilingual-e5-base` 或 `sentence-transformers/paraphrase-multilingual-MiniLM-L12-v2`。
- 检索时使用“中文原问题 + 术语英译扩展”的查询策略，例如把“交叉验证”扩展为 “cross-validation”。
- 回答生成使用可配置的大模型 API，优先适配 OpenAI-compatible 接口；可接通义千问、智谱或其他可用服务。

为什么不默认 Neo4j：

- 本题评分要求的是知识图谱辅助问答和可视化，不要求必须使用图数据库。
- 课程项目的图谱规模较小，SQLite + NetworkX 足够完成路径、邻居和子图查询。
- Neo4j 会增加安装、连接、部署和报告复现成本；除非老师明确要求，否则工程收益不高。

## 数据构建约束

问答对字段至少包含：

- question：中文问题。
- answer：中文答案。
- type：概念解释、关系查询、应用场景。
- concept_ids：关联知识点。
- source_refs：教材章节、页码或图谱来源。
- quality_status：草稿、已校对、需复核。

知识点字段至少包含：

- name_cn：中文名称。
- name_en：英文名称。
- aliases：别名。
- chapter：主要来源章节。
- description：一句话说明。

图谱边字段至少包含：

- source_concept。
- target_concept。
- relation_type。
- evidence：关系依据或来源。

RAG 文本块字段至少包含：

- chunk_id。
- text。
- chapter。
- section。
- pdf_page。
- source_file。
- embedding_model。

## 回答质量约束

系统回答必须遵守：

- 用简体中文回答。
- 优先依据检索到的教材片段和知识图谱关系，不编造教材外结论。
- 对概念解释类问题，给出定义、直观解释、适用条件和相关概念。
- 对关系查询类问题，给出相同点、不同点、适用场景，并展示图谱关系或子图。
- 对应用场景类问题，给出适合/不适合的场景和简短例子。
- 当检索证据不足时，明确说明“当前教材证据不足”，并给出可继续查询的相关概念。
- 每个 RAG 回答展示 3-5 条来源，至少包含教材文件、章节或页码。

## 页面与交互约束

核心页面应包含：

- 登录/注册页。
- 问答工作台：问题输入、回答、来源、相关问题、图谱子图、历史入口。
- 问答对管理页：列表、搜索、筛选、新增、编辑、删除。
- 知识图谱页：全图浏览、节点点击、邻域展开、问题推荐。
- 历史记录页：按用户展示问题、时间、答案摘要，支持清空。

界面应以课程学习工具为定位：清晰、克制、信息密度适中，不做过度装饰。

## 验收清单

后续目标模式完成时，至少要能证明：

- 系统能从英文教材 PDF 抽取文本并建立向量库。
- 向量库中文问题检索能命中英文教材中的相关章节。
- 至少 50 个问答对存在于系统中，并可通过管理界面 CRUD。
- 关系查询问题能展示知识图谱路径或核心概念子图。
- 点击图谱节点能推荐 3 个相关问题。
- 用户注册、登录、历史记录和清空功能可用。
- 至少 6 个代表性问题通过端到端演示：概念解释 2 个、关系查询 2 个、应用场景 2 个。
- 源码有 README 或运行说明，实验报告完整。
- 本地启动命令经过实际验证。

## 默认不做的事

- 不训练新的大模型。
- 不默认 OCR 中文扫描教材。
- 不默认使用 Neo4j、LangChain 或复杂微服务，除非后续证明确有必要。
- 不把大模型 API Key 写入代码或提交到仓库。
- 不为了炫技牺牲课程项目的可复现性。

## 需要用户确认或准备

1. 教材主语料：建议确认使用英文 `ISLRv2_corrected_June_2023.pdf` 作为主语料，中文教材仅作为人工术语参考。
2. 大模型 API：后续需要提供或配置至少一个可用 API Key，例如通义千问、智谱或其他 OpenAI-compatible 服务。
3. 作业一知识图谱：如果已有作业一产物，请放入当前目录；如果没有，则后续目标模式默认手动构建一份满足题目要求的课程知识图谱。
4. 上线口径：默认交付本地可运行演示版和清晰运行文档；若需要公网访问，需要再选择部署平台。
