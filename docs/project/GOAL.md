# 下一阶段目标清单

本文用于下一轮开启 goal 目标模式时作为任务契约。当前轮只做仓库阅读、问题整理和目标收敛，不做实际功能实现。

## 总目标

把当前已经可运行、已上线的“高级统计方法知识图谱智能问答系统”继续打磨到更符合题目原始要求、更适合课程验收、更适合长期维护的状态。最终交付应同时满足本地验证、真实浏览器视觉验收和 VPS 公网可用。

## 当前基线

- 公网地址：`https://aistudyassistant.bluesclawd.dev`。
- 自动部署：`main` 分支 push 后由 GitHub Actions 上传到 VPS 并重建 Docker 服务。
- 当前数据规模：69 个知识点、336 条图谱边、70 个问答对、1413 个教材文本块。
- 当前 RAG 主语料：`ISLRv2_corrected_June_2023.pdf`。
- 当前检索实现：`backend/app/rag.py` 中的本地 TF-IDF / 术语扩展式混合检索，`TextChunk.embedding_model` 当前标记为 `local-clean-v3` 或 `local-hybrid`，还不是真正的向量数据库。
- 当前前端：React 单页应用，核心界面集中在 `frontend/src/main.tsx`，管理后台和问答对管理存在列表过长、信息架构偏拥挤的问题。
- 当前重要异常：`docs/架构与功能说明.md` 在工作区内显示为 0 字节且处于 modified 状态，下一阶段实际动手前必须先确认是否为用户未完成改动，不能擅自覆盖。

## P0：向量数据库 RAG 与可切换检索架构

1. 实现符合题目原文要求的向量数据库 RAG 流程：教材分块、文本向量化、向量索引持久化、问题向量检索、返回 3-5 个相关教材块。
2. 保留现有 TF-IDF / 本地混合检索作为轻量降级路径，避免 VPS 性能或依赖问题导致系统不可用。
3. 设计统一检索接口，使后端可以通过环境变量或管理员配置在 `tfidf`、`vector`、`hybrid` 等模式间切换。
4. 向量模式优先考虑 VPS 资源可承受的方案：FAISS 或 Chroma 均可评估；默认应选择依赖可控、镜像构建稳定、内存开销较低的实现。
5. 保留并复用现有可信度守门逻辑：`out_of_scope`、`insufficient_evidence`、无关问题空来源/空推荐/空图谱、多轮追问重新检索等不变量不能回退。
6. 在管理后台和系统统计中展示当前检索模式、向量索引状态、索引更新时间、文本块数量和重建结果。

## P0：管理后台与问答对管理的信息架构重构

1. 问答对管理不能把 70 条问答对全部无分页铺满页面，应增加分页、搜索、类型筛选、知识点筛选、质量状态筛选和清晰的编辑态。
2. 管理后台不应继续把用户、模型配置、问答对、文本块、知识点、图谱边、历史审计全部堆在一个长页面里。
3. 推荐把管理后台拆成总览 + 子模块：
   - 总览：系统状态、检索模式、LLM 状态、教材状态、最近维护动作。
   - 用户与权限：用户、角色、改密、注销、删除。
   - 教材与索引：参考书管理、文本块搜索、索引重建、向量库状态。
   - 问答对：问答对 CRUD、筛选、分页、质量状态。
   - 知识点图谱：知识点 CRUD、图谱边 CRUD、关系检索。
   - 历史审计：提问历史、回答状态、来源回看。
4. “系统状态”中的“知识点图谱”“问答对”“文本块”等指标应可点击进入对应子页面或子视图。
5. 子页面可以先在单页应用内用二级导航实现，不必马上引入 React Router；但状态、URL 或至少页面内导航体验应足够清晰。

## P0：管理员可配置 RAG 参考书

1. 在管理员页面增加参考书管理能力，使管理员能查看当前 RAG 教材、上传或指定新的 PDF，并触发重建索引。
2. 设计教材元数据模型，至少记录文件名、显示名称、是否启用、页数、文本块数量、索引模式、索引状态、创建时间和更新时间。
3. 当前 `APP_PDF_PATH` 仍作为默认启动语料，但运行时应能选择新的参考书作为 RAG 主语料。
4. 新教材重建索引必须有明确状态：待处理、处理中、成功、失败；失败时保留错误信息，不应破坏旧索引。
5. 安全边界：只允许管理员操作；上传文件大小、类型、路径必须受控；不能允许任意路径读取服务器文件。
6. 部署边界：上传教材和向量索引属于运行数据，应保存在 `data/` 或 Docker volume 中，不提交 Git。

## P1：前端体验与视觉质量系统性改进

1. 以课程学习工具和管理控制台为定位，优先优化信息层级、可扫描性、分页、空状态、加载态、错误态和移动端可用性。
2. 使用 Design Taste Frontend 作为审美审查参考，但该 skill 明确更偏 landing page / redesign；本项目是学习工具和管理后台，应以产品 UI、数据表格、表单和工作台体验为主，不追求营销页风格。
3. 重点审查：
   - 问答工作台是否在长回答、多来源、图谱、推荐问题同时出现时仍清晰。
   - 管理后台是否可快速定位任务，而不是滚动寻找。
   - 问答对、文本块、历史审计是否使用分页或虚拟化，避免长列表拖慢页面。
   - 表格、卡片、按钮、标签、输入框是否有一致的尺寸、间距和状态。
   - 移动端是否不存在文字溢出、按钮挤压、图谱遮挡和横向滚动失控。
4. 保留现有 Cytoscape.js 图谱，不恢复手写 SVG 图谱。
5. 保留中文默认界面和 Markdown/KaTeX 回答渲染。

## P1：工程结构、性能与可维护性

1. `frontend/src/main.tsx` 已经承载过多组件，下一阶段可按页面或领域拆分组件，但应保持改动可验证、避免一次性无边界重写。
2. PDF 索引重建目前是同步接口，下一阶段应评估后台任务化或至少前端进度提示，避免大文件重建时请求阻塞。
3. 向量索引、文本块、教材元数据、检索模式之间需要清晰边界，避免把所有逻辑继续塞进 `rag.py`。
4. 若引入新依赖，必须评估 Docker 镜像体积、VPS 内存、构建时间和 GitHub Actions 自动部署稳定性。
5. 所有新增配置必须同步 `.env.example`、README、运维和架构文档。

## P1：深度审查、测试与上线

1. 完成所有功能后运行 `.\scripts\verify.ps1`，并根据新增功能扩展后端测试和 Playwright 覆盖。
2. 页面改动必须用 Playwright 生成并人工检查桌面端和移动端真实截图，覆盖问答工作台、问答对分页、管理总览、教材/索引、知识点图谱、文本块、用户管理、历史/笔记。
3. RAG 改动必须加入检索质量样例，对比 TF-IDF 与向量模式在代表性中文问题上的命中结果。
4. 权限审查必须覆盖普通学生无法访问上传教材、重建索引、用户管理、问答对写入、知识点/图谱边写入等后台能力。
5. 线上发布前必须本地验证通过；发布后必须检查 `https://aistudyassistant.bluesclawd.dev/api/health`、`/api/system/stats`、公网登录、无关问题拒答、课程问题回答和管理后台关键页。
6. 最终需要更新 `docs/对抗性审查.md`、`docs/验收记录.md`、`docs/架构与功能说明.md`、`README.md`、`docs/运维手册.md` 和必要的实验报告/交付物说明。
